# ADR-002: Shared Domain Contracts & Type Strategy

## Context

JobPilot is a TypeScript monorepo with a NestJS backend and a React frontend
that must agree on data structure. We want one source of truth for the domain
model (no drift between FE and BE), runtime validation at the API boundary, and
correct handling of values that don't survive JSON cleanly (notably `Date`,
which becomes a string on the wire).

## Decision

1. **Single shared package.** All cross-boundary contracts live in
   `packages/shared-types`, consumed by both apps via `workspace:*`.

2. **zod-first everywhere; types are inferred.**
   - **Shared building blocks** (`src/utils.ts`, `src/common.ts`): branded ID
     schemas, `isoDate`, `Salary`, `Timestamps`.
   - **Domain models** (`src/domain/*`): each entity is a zod schema; its TS
     type is `z.infer`'d. The schema is the source of truth.
   - **API DTOs** (`src/dto/*`): **projections of the domain schema** via
     `.pick()` / `.omit()` / `.partial()` / `.extend()`. Because DTOs are
     derived from the domain schema, they cannot drift from the domain model.

3. **Branded IDs** via `z.uuid().brand<...>()` for nominal type safety.

4. **Date boundary.** Domain types resolve to `Date`. On the wire, dates are ISO
   strings; `isoDate = z.iso.datetime({ offset: true }).pipe(z.coerce.date())`
   validates the string and coerces to `Date` (input `string`, output `Date`).
   zod parses string → `Date` when validating request bodies (backend) and
   responses (frontend). Rule: validate/`.parse()` at boundaries, never cast.

5. **Persistence-only fields stay on the backend entity**, never in the shared
   contract — e.g. the pgvector embedding on `CvProfile`, password hashes, etc.

### Notes / patterns

- The `Application` schema is an **intersection** of a base object and a
  `z.discriminatedUnion('status', …)`. Intersections lack `.pick/.omit`, so the
  base fields are also exported as `baseApplicationSchema` for DTOs to derive
  from.
- URL/email fields use `z.url()` / `z.email()`; opaque references that may be
  storage keys (e.g. `cvProfile.fileUrl`, `company.logo`) stay `z.string()`.

## Consequences (Trade-offs)

- **Pros:** one source of truth (no FE/BE drift); DTOs provably derive from the
  domain (`.pick/.omit`); runtime validation at the edge with zod; honest date
  handling across the wire; nominal ID safety.
- **Cons:** `shared-types` ships runtime code (a `zod` dependency on both FE and
  BE), not just types; the domain layer is no longer dependency-free.

## Status

Accepted. Supersedes the earlier hybrid (type-first domain + zod-first DTO)
proposal — derived DTOs require a runtime domain schema, so the domain is
zod-first too.
